# VR Test Determinism Fix

> **For agentic workers:** Read this plan in full before touching code. The
> problem is subtle ‚Äî previous attempts at a fix (constant `Math.random`)
> did NOT solve it. The real fix likely requires seeding IndexedDB or
> intercepting at the module level, not patching `Math.random` via
> `addInitScript`. Work on a branch off master in a new worktree.

## Background

VR tests under `e2e/visual.spec.ts` are non-deterministic between runs.
Round values, tile values, and distractor ordering drift every time
something upstream changes the rate at which `Math.random()` is consumed,
causing baseline PNGs to mismatch.

### Symptoms observed (2026-04-10)

1. Baseline `number-match-numeral-to-group-dark-chromium.png` previously
   showed the numeral `8`; after unrelated Wave 2 work it rendered `10`.
2. After changing `e2e/seed-math-random.ts` from a Lehmer PRNG to a
   constant `() => 0.37`:
   - `yarn test:vr:update` regenerated baselines with the new constant.
   - `yarn test:vr` (rerun IMMEDIATELY after update) still flagged 7
     baselines as mismatched ‚Äî including tests that weren't failing in
     the first pass.
   - Inspection of
     `test-results/‚Ä¶/number-match-numeral-to-group-dark-diff.png` showed
     the expected rendered `4` while the actual rendered `8`. Neither
     value matches what `Math.floor(0.37 √ó 12) + 1 = 5` should produce.
3. Conclusion: the init-script `Math.random` override is **not in effect**
   when the React route loader calls `makeDefaultNumberMatchConfig()`, OR
   something else besides `Math.random` is non-deterministic.

## Prior attempts

### Attempt 1: Lehmer PRNG (current state on master)

`e2e/seed-math-random.ts` installs a seeded Lehmer PRNG via
`page.addInitScript`. This gives a deterministic sequence but is
sensitive to **sequence-offset drift** ‚Äî any upstream change that
consumes `Math.random` one more time before the round generator runs
shifts every downstream value. This is brittle and is the direct cause
of the "8 ‚Üí 10" baseline drift reported above.

### Attempt 2: Constant (`() => 0.37`) ‚Äî tried in `feat/countable-dots` worktree

Intended to collapse every `Math.random()` call to the same output,
making sequence-offset irrelevant.

**Result: did not stabilise the tests.** Baselines regenerated by
`yarn test:vr:update` still mismatched on the very next `yarn test:vr`
run, with values (4, 8) that don't match `Math.floor(0.37 √ó 12) + 1 = 5`.

This strongly suggests the `addInitScript` override is being **bypassed**
or **applied too late** for the React route loader. Possible reasons:

1. The loader runs inside a deferred async context that captures
   `Math.random` from a snapshot taken before init scripts evaluated.
2. A service worker or prerendered shell is serving cached JS that
   consumed `Math.random` before the init script patched it.
3. Tanstack Router loaders resolve during a microtask that precedes
   `addInitScript` execution on subsequent navigations (`goto` after
   settings panel open).
4. Storage from previous tests is leaking ‚Äî IndexedDB may be persisting
   across test contexts via the mounted Docker volume or Playwright's
   default storage isolation may not cover the game's Dexie DB.

## Files involved

- `e2e/seed-math-random.ts` ‚Äî current `Math.random` pin helper
- `e2e/visual.spec.ts` ‚Äî all VR tests, calls `seedMathRandom` in
  `test.beforeEach`
- `scripts/vr-docker.mjs` ‚Äî Docker runner that mounts `${cwd}:/work`
- `src/routes/$locale/_app/game/$gameId.tsx` ‚Äî contains
  `makeDefaultNumberMatchConfig()` with the problematic
  `Math.random`-based round generation at lines 119‚Äì122 and the
  `resolveNumberMatchConfig` resolver at lines 185‚Äì204
- `src/games/number-match/build-numeral-round.ts` ‚Äî contains
  `shuffleInPlace` that uses `Math.random` for distractor ordering
- `src/games/build-round-order.ts` ‚Äî `buildRoundOrder` shuffles round
  order when `roundsInOrder: false` (the default)

## Recommended approach

### Option A (preferred): Seed IndexedDB directly

Bypass `Math.random` entirely by writing a known fixed NumberMatch config
into IndexedDB (Dexie) **before** the page navigates, via
`page.addInitScript`. This removes all randomness from the test path:
the loader reads the saved config as-is, and `resolveNumberMatchConfig`
returns those fixed values.

Steps:

1. Find the Dexie DB name and the settings/config store ‚Äî likely under
   `src/db/`. Identify the record shape for game configs.
2. In `seed-math-random.ts` (or a new `seed-vr-fixtures.ts`), add an
   `addInitScript` that opens the DB and `put`s fixed configs for
   every game used in VR tests:

   ```typescript
   await page.addInitScript(() => {
     // Open Dexie DB and write fixed game configs
     const request = indexedDB.open('base-skill' /* version */);
     request.onsuccess = () => {
       const db = request.result;
       const tx = db.transaction('settings', 'readwrite');
       tx.objectStore('settings').put(
         {
           id: 'game-config:number-match',
           gameId: 'number-match',
           component: 'NumberMatch',
           mode: 'numeral-to-group',
           tileStyle: 'dots',
           tileBankMode: 'distractors',
           distractorCount: 2,
           totalRounds: 3,
           roundsInOrder: true, // ‚Üê force deterministic order
           rounds: [{ value: 5 }, { value: 8 }, { value: 3 }],
           range: { min: 1, max: 12 },
           ttsEnabled: true,
           inputMethod: 'drag',
           wrongTileBehavior: 'lock-auto-eject',
         },
         'game-config:number-match',
       );
     };
   });
   ```

3. Set `roundsInOrder: true` in the fixture so `buildRoundOrder` returns
   sequential indices ‚Äî no shuffle, no randomness.
4. For tile distractor ordering: either keep a constant `Math.random`
   pin alongside the DB seed, OR refactor `buildNumeralRound` to accept
   an optional `shuffleFn` parameter that tests can pin.
5. Apply the same pattern for `word-spell` and `sort-numbers` so those
   VR tests are also stable.
6. Regenerate all baselines: `yarn test:vr:update` in Docker (must have
   Docker running locally, per `CLAUDE.md` and the memory feedback).
7. Run `yarn test:vr` twice in a row to verify stability.

### Option B: Test-only URL query param

Add a gated query param handler like `?vr-fixture=<game-id>` that the
route loader recognises and substitutes for the real resolver. Example:

```typescript
const resolveNumberMatchConfig = (
  saved: Record<string, unknown> | null,
) => {
  if (
    import.meta.env.DEV ||
    (typeof globalThis !== 'undefined' &&
      new URLSearchParams(globalThis.location?.search).get(
        'vr-fixture',
      ) === 'number-match')
  ) {
    return VR_NUMBER_MATCH_FIXTURE;
  }
  // ...
};
```

Downside: leaks test affordance into production code paths. Guarded by
`import.meta.env.DEV` but still a smell.

### Option C: Mock `Math.random` at the module level via Vite plugin

Add a `vite.config.ts` mode that, when run under Playwright, replaces
`Math.random` in the bundled output. Complex and fragile. Not recommended.

## Verification checklist

When a fix is applied:

- [ ] `yarn test:vr` passes on first run after
      `yarn test:vr:update` (no drift).
- [ ] Running `yarn test:vr` three times in a row with no code changes
      produces zero baseline mismatches.
- [ ] A trivial unrelated code change (e.g. adding a log in the route
      loader) does NOT shift any baseline.
- [ ] The fix is documented in the PR body with the root-cause
      explanation so future agents don't revisit the same dead end.
