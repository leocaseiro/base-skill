## 1. Overview

BaseSkill is a free, open-source, offline-first educational PWA for children. It supports multiple languages via **react-i18next**, enabling each child profile to operate in its own preferred language independently.

**Initial supported languages:**

| Code    | Language               | Status  |
| ------- | ---------------------- | ------- |
| `en`    | English (Australian)   | Primary |
| `pt-BR` | Portuguese (Brazilian) | Initial |

**Key principles:**

- Language is a **per-profile setting**, not a global app setting. Two children on the same device can use different languages simultaneously (each in their own profile session).
- No server-side i18n is required. BaseSkill is a static SPA with all translations bundled locally for offline use.
- The architecture is extensible — adding a new language requires no changes to core logic, only new locale files and content mappings.
- All translation keys are namespaced to reduce bundle size and support lazy-loading.

## 2. i18n Framework Choice: react-i18next

### Why react-i18next

- Most widely adopted React i18n library (>10M weekly downloads)
- Excellent React integration via `useTranslation` hook and `Trans` component
- Namespace support for logical code splitting of translation bundles
- Lazy-loading of translation namespaces via `i18next-resources-to-backend` or dynamic imports
- Compatible with TanStack Router (no framework-level i18n required)
- TypeScript-first: supports generated type definitions from translation files
- Supports pluralization, interpolation, and context-based translations out of the box

### Configuration: `app/i18n/i18n.ts`

```typescript
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';

// Namespaces are loaded statically for offline-first support.
// All locale files are bundled with the app.
import enCommon from './locales/en/common.json';
import enGames from './locales/en/games.json';
import enSettings from './locales/en/settings.json';
import enEncouragements from './locales/en/encouragements.json';
import enErrors from './locales/en/errors.json';

import ptBRCommon from './locales/pt-BR/common.json';
import ptBRGames from './locales/pt-BR/games.json';
import ptBRSettings from './locales/pt-BR/settings.json';
import ptBREncouragements from './locales/pt-BR/encouragements.json';
import ptBRErrors from './locales/pt-BR/errors.json';

export const SUPPORTED_LOCALES = ['en', 'pt-BR'] as const;
export type SupportedLocale = (typeof SUPPORTED_LOCALES)[number];
export const DEFAULT_LOCALE: SupportedLocale = 'en';

i18n.use(initReactI18next).init({
  // Language is set at runtime from profile settings, not from browser/OS.
  // initImmediate ensures synchronous init on first render.
  lng: DEFAULT_LOCALE,
  fallbackLng: DEFAULT_LOCALE,
  ns: ['common', 'games', 'settings', 'encouragements', 'errors'],
  defaultNS: 'common',
  interpolation: {
    // React already escapes values — no double-escaping needed.
    escapeValue: false,
  },
  resources: {
    en: {
      common: enCommon,
      games: enGames,
      settings: enSettings,
      encouragements: enEncouragements,
      errors: enErrors,
    },
    'pt-BR': {
      common: ptBRCommon,
      games: ptBRGames,
      settings: ptBRSettings,
      encouragements: ptBREncouragements,
      errors: ptBRErrors,
    },
  },
});

export default i18n;
```

### Language Detection: Profile-Driven, Not Browser-Driven

Language detection from the browser or OS is **explicitly disabled**. Language is always sourced from the active profile's `settings.activeLanguage` field in RxDB. When a profile is activated, call:

```typescript
import i18n from '~/i18n/i18n';

export async function applyProfileLanguage(
  language: SupportedLocale,
): Promise<void> {
  await i18n.changeLanguage(language);
}
```

This function is called on:

- Profile selection (profile picker screen)
- Profile switch during a session
- App startup after restoring the last active profile from RxDB

## 3. TanStack Router Locale Integration

### Language is Profile-Scoped, Not URL-Scoped

BaseSkill does **not** use locale-based URL segments (e.g., no `/en/dashboard` or `/pt-BR/games`). The language belongs to the profile, not the URL. This avoids URL churn when switching profiles and simplifies routing.

### Router Context: `currentLanguage`

TanStack Router's context is used to make the active language available to all loaders and route components in a type-safe way.

**`app/router.tsx`:**

```typescript
import type { SupportedLocale } from '~/i18n/i18n';

export interface RouterContext {
  currentLanguage: SupportedLocale;
  // ... other context fields (queryClient, activeProfileId, etc.)
}
```

**Updating context on language change:**

```typescript
// Called after i18n.changeLanguage() resolves
router.invalidate(); // triggers loaders to re-run with updated context
```

**Using language in a loader:**

```typescript
export const Route = createFileRoute('/games/')({
  loader: async ({ context }) => {
    const { currentLanguage } = context;
    const gameContent = await loadGameContent(currentLanguage);
    return { gameContent };
  },
});
```

### Profile Switch Flow

```
User taps profile → ProfilePicker
  → reads profile.settings.activeLanguage from RxDB
  → calls i18n.changeLanguage(language)
  → updates RouterContext.currentLanguage
  → router.invalidate() (loaders re-run)
  → navigate to /dashboard
```

## 4. Translation File Structure

All locale files live under `app/i18n/locales/`. Each language has five namespaces:

```
app/i18n/locales/
  en/
    common.json          # App-wide strings: nav, buttons, status messages
    games.json           # Game titles, categories, subject names, instructions
    settings.json        # Settings labels and descriptions
    encouragements.json  # Contextual encouragement messages (arrays)
    errors.json          # Error messages
  pt-BR/
    common.json
    games.json
    settings.json
    encouragements.json
    errors.json
```

### Namespace Responsibilities

| Namespace        | Loaded on           | Contents                                                      |
| ---------------- | ------------------- | ------------------------------------------------------------- |
| `common`         | App init (always)   | Nav, buttons, status, profile management                      |
| `games`          | Game routes         | Titles, categories, subjects, instructions, grade band labels |
| `settings`       | Settings route      | Form labels, option names, descriptions                       |
| `encouragements` | Active game session | Arrays of encouragement messages per outcome                  |
| `errors`         | App init (always)   | User-facing error messages                                    |

## 6. Language-Specific Game Content Strategy

### Overview

Game configurations carry a `content` map keyed by locale. This separates game mechanics (language-neutral) from game content (language-specific).

```typescript
type GameContent = {
  words?: string[];
  sentences?: string[];
  letters?: string[];
};

type GameConfig = {
  id: string;
  type: GameType;
  gradeBand: GradeBand;
  content: Partial<Record<SupportedLocale, GameContent>>;
};
```

### Content by Game Type

#### Math Games (Counting, Addition, Subtraction)

Math game content is **language-neutral** — number operations are universal. Only UI instructions and labels are localized via `games.json`. No per-locale `content` entries are needed.

```typescript
const countingGame: GameConfig = {
  id: 'counting-prек-1',
  type: 'counting',
  gradeBand: 'preK',
  content: {}, // empty — math content generated programmatically
};
```

#### Word Builder

Word lists are locale-specific. Each locale provides 3–6 letter words appropriate for the grade band.

```typescript
const wordBuilderGame: GameConfig = {
  id: 'word-builder-kindergarten-1',
  type: 'wordBuilder',
  gradeBand: 'kindergarten',
  content: {
    en: {
      words: [
        'cat',
        'dog',
        'hat',
        'sun',
        'run',
        'big',
        'fun',
        'map',
        'cup',
        'hop',
      ],
    },
    'pt-BR': {
      words: [
        'gato',
        'casa',
        'bola',
        'mão',
        'rio',
        'sol',
        'pé',
        'mar',
        'lua',
        'céu',
      ],
    },
  },
};
```

#### Read Aloud

Sentences are locale-specific, grade-leveled, and written to match reading comprehension expectations per language.

```typescript
const readAloudGame: GameConfig = {
  id: 'read-aloud-year1-1',
  type: 'readAloud',
  gradeBand: 'year1to2',
  content: {
    en: {
      sentences: [
        'The cat sat on the mat.',
        'I can see a big red bus.',
        'The dog runs fast in the park.',
      ],
    },
    'pt-BR': {
      sentences: [
        'O gato está no tapete.',
        'Eu vejo um ônibus vermelho grande.',
        'O cachorro corre rápido no parque.',
      ],
    },
  },
};
```

#### Letter Tracing

Letter Tracing supports the Latin alphabet, which is shared between `en` and `pt-BR`. Both languages use the same 26-letter set plus accented characters for `pt-BR`.

```typescript
const letterTracingGame: GameConfig = {
  id: 'letter-tracing-prek-a',
  type: 'letterTracing',
  gradeBand: 'preK',
  content: {
    en: {
      letters: ['A', 'B', 'C', 'D', 'E', 'F'],
    },
    'pt-BR': {
      letters: ['A', 'B', 'C', 'D', 'E', 'F'],
    },
  },
};
```

For `pt-BR`-specific sessions, accented letters (Ã, Ç, Ê, etc.) can be introduced at higher grade bands.

### Content Fallback

If content is not available in the profile's active language, fall back to `en`:

```typescript
function getGameContent(
  game: GameConfig,
  locale: SupportedLocale,
): GameContent {
  return game.content[locale] ?? game.content['en'] ?? {};
}
```

## 7. TTS Voice Selection

### Overview

Parents select a TTS voice per language per profile in **Parent Settings > Voice Selector**. Voice availability is device-dependent (`speechSynthesis.getVoices()` returns different results per OS and browser).

### Voice Enumeration

```typescript
function getVoicesForLanguage(
  language: SupportedLocale,
): SpeechSynthesisVoice[] {
  const langPrefix = language === 'en' ? 'en-' : language;
  return speechSynthesis
    .getVoices()
    .filter((v) => v.lang.startsWith(langPrefix));
}
```

Voices are enumerated on app load. On some platforms (Chrome on Android), voices load asynchronously — use the `voiceschanged` event:

```typescript
speechSynthesis.addEventListener('voiceschanged', () => {
  // Re-enumerate and update voice picker UI
});
```

### Default Voice Selection

```typescript
const DEFAULT_VOICE_PREFERENCES: Record<SupportedLocale, string[]> = {
  en: ['en-AU', 'en-GB', 'en-US'], // prefer Australian accent
  'pt-BR': ['pt-BR', 'pt-PT'], // prefer Brazilian Portuguese
};

function getDefaultVoice(
  language: SupportedLocale,
): SpeechSynthesisVoice | null {
  const voices = getVoicesForLanguage(language);
  const preferences = DEFAULT_VOICE_PREFERENCES[language];
  for (const preferred of preferences) {
    const match = voices.find((v) => v.lang === preferred);
    if (match) return match;
  }
  return voices[0] ?? null;
}
```

### Voice Picker UI

The voice selector in Parent Settings displays:

- Language group headers
- Voice name + language tag (e.g., "Karen — en-AU")
- A **Preview** button that plays the `settings.voiceSelector.sampleText` translation in the selected voice
- A note that voice availability depends on the device

```typescript
type VoicePickerEntry = {
  voice: SpeechSynthesisVoice;
  displayName: string; // e.g., "Karen — en-AU"
  isDefault: boolean;
};
```

### Voice Preference Storage in RxDB

Voice preferences are stored per profile, per language, and tagged with the current `deviceId` (a stable UUID generated on first app install):

```typescript
type VoicePreference = {
  profileId: string;
  language: SupportedLocale;
  voiceName: string;
  deviceId: string; // UUID — voices differ per device/OS
};
```

**On a new/different device:** The stored `voiceName` may not exist. The resolver falls back to `getDefaultVoice(language)`:

```typescript
function resolveVoice(
  pref: VoicePreference | null,
  currentDeviceId: string,
  language: SupportedLocale,
): SpeechSynthesisVoice | null {
  if (pref && pref.deviceId === currentDeviceId) {
    const voices = getVoicesForLanguage(language);
    const match = voices.find((v) => v.name === pref.voiceName);
    if (match) return match;
  }
  return getDefaultVoice(language);
}
```

## 8. STT Language Matching

### Overview

`SpeechRecognition.lang` must be set to match the profile's active language. The recognition lang uses BCP-47 tags.

### Language Code Mapping

```typescript
const STT_LANG_MAP: Record<SupportedLocale, string> = {
  en: 'en-AU', // default to Australian English for STT
  'pt-BR': 'pt-BR',
};

function getSttLang(language: SupportedLocale): string {
  return STT_LANG_MAP[language];
}
```

### STT Initialization

```typescript
function createSpeechRecognition(
  language: SupportedLocale,
): SpeechRecognition | null {
  const SpeechRecognitionAPI =
    window.SpeechRecognition ?? window.webkitSpeechRecognition;

  if (!SpeechRecognitionAPI) return null;

  const recognition = new SpeechRecognitionAPI();
  recognition.lang = getSttLang(language);
  recognition.interimResults = false;
  recognition.maxAlternatives = 3;
  return recognition;
}
```

### Fallback to Text Input

If STT is unavailable for the selected language (or the API is not supported), the game falls back gracefully to a text input field. The `errors.sttUnavailable` translation key is shown to the user.

```typescript
function isSttAvailable(language: SupportedLocale): boolean {
  const SpeechRecognitionAPI =
    window.SpeechRecognition ?? window.webkitSpeechRecognition;
  if (!SpeechRecognitionAPI) return false;
  // Further check: attempt recognition and listen for onerror 'language-not-supported'
  return true;
}
```

## 10. Adding a New Language

Follow these steps to add a new language (e.g., Spanish — `es`):

### Step 1: Create locale files

Create the directory and all five namespace files:

```
app/i18n/locales/es/
  common.json
  games.json
  settings.json
  encouragements.json
  errors.json
```

Copy `en/` versions as a translation baseline. Translate all values. Do **not** translate JSON keys.

### Step 2: Add game content entries

For each game config that has locale-keyed `content`, add an `es` entry:

```typescript
content: {
  en: { words: ['cat', 'dog'] },
  'pt-BR': { words: ['gato', 'cachorro'] },
  es: { words: ['gato', 'perro'] },  // ← add this
}
```

Math game configs require no change (content is language-neutral).

### Step 3: Add locale to `SUPPORTED_LOCALES`

In `app/i18n/i18n.ts`:

```typescript
export const SUPPORTED_LOCALES = ['en', 'pt-BR', 'es'] as const;
```

Import and register the new locale's JSON files in the `i18n.init()` resources object.

### Step 4: Add TTS/STT language code mapping

In the TTS default voice preferences:

```typescript
const DEFAULT_VOICE_PREFERENCES: Record<SupportedLocale, string[]> = {
  en: ['en-AU', 'en-GB', 'en-US'],
  'pt-BR': ['pt-BR', 'pt-PT'],
  es: ['es-ES', 'es-MX', 'es-AR'], // ← add this
};
```

In the STT language mapping:

```typescript
const STT_LANG_MAP: Record<SupportedLocale, string> = {
  en: 'en-AU',
  'pt-BR': 'pt-BR',
  es: 'es-ES', // ← add this
};
```

### Step 5: Add grade naming convention

Translate the `gradeBands` section in `app/i18n/locales/es/games.json` using the local education system's terminology.

### Step 6: Run i18n snapshot tests

```bash
pnpm test:i18n
```

This runs the i18n key completeness check (see §11). All keys present in `en` must be present in `es`. Fix any missing keys before merging.

### Step 7: (Optional) Add to Playwright E2E smoke test

In `e2e/i18n.spec.ts`, add `es` to the locale smoke test array:

```typescript
const SMOKE_LOCALES: SupportedLocale[] = ['en', 'pt-BR', 'es'];
```

The smoke test creates a profile in each language, navigates to a game, and asserts that UI strings render in the correct language.

## 11. Missing Translation Handling

### Runtime Fallback

react-i18next automatically falls back to `fallbackLng: 'en'` when a key is missing in the active language. No UI breakage occurs — the English string is shown silently.

### Development Warning

In development mode (`import.meta.env.DEV`), react-i18next logs a console warning when a key is missing in the active language:

```
i18next: key "games:wordBuilder.someNewKey" not found in "pt-BR", using fallback language "en"
```

This is enabled by the default `missingKeyHandler` in react-i18next. No additional configuration is needed.

### CI Completeness Lint

A CI step checks that every key present in the `en` locale exists in all other supported locales. This runs as a dedicated script, not as part of the app bundle:

```typescript
// scripts/check-i18n-completeness.ts
import { SUPPORTED_LOCALES } from '../app/i18n/i18n';

const NAMESPACES = [
  'common',
  'games',
  'settings',
  'encouragements',
  'errors',
];

function flattenKeys(
  obj: Record<string, unknown>,
  prefix = '',
): string[] {
  return Object.entries(obj).flatMap(([key, value]) => {
    const fullKey = prefix ? `${prefix}.${key}` : key;
    if (Array.isArray(value)) return [fullKey]; // array values are leaf nodes
    if (typeof value === 'object' && value !== null) {
      return flattenKeys(value as Record<string, unknown>, fullKey);
    }
    return [fullKey];
  });
}

// For each namespace, compare en keys against all other locales.
// Exit with code 1 if any keys are missing (fails CI).
```

Run via:

```bash
pnpm i18n:check
```

Add to CI pipeline (`ci.yml`) as a required check before merge.
