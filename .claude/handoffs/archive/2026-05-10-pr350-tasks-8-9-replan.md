## Critical findings the new plan must resolve

Quoting from the review doc, in priority order:

**C1 тАФ Round-advance is structurally broken**

- Guards (`isMidLevelRound`, `isLastRoundOfLevel`, `isLastRound`) are
  declared as `() => false` placeholders in Task 8's `setup()`. Plan
  claims `useGameEngine` injects real guards via `provide()`, but Task 7
  doesn't show this wiring. **Both `NEXT` transitions guarded тЖТ `NEXT`
  silently dropped тЖТ round 2 never starts.**
- `waitingForNext.NEXT` actions list is `['incrementRoundIndex',
'buildRound', 'advanceRound']`. `buildRound` is no-op (Spec Delta #6),
  doesn't `assign` to `lastRoundOutput`. `advanceRound` reads
  `context.lastRoundOutput`, never populated. Adapter throws "missing {
  tiles, zones }". **Round advance throws at runtime every round.**

Resolution direction: drop `'buildRound'`/`'advanceRound'` from the
`NEXT` actions for NumberMatch (component drives `ADVANCE_ROUND`
directly per Spec Delta #6's premise), keep only
`'incrementRoundIndex'`. AND inject real guard bodies in Task 7 with
explicit prescription.

**C2 тАФ `gameOver` broken three ways**

- `type: 'final'` + 60s `gameOverActor` timer тЖТ `completeGame` runs 60
  seconds after game-over screen appears.
- No `CELEBRATION_DONE` early-dismiss handler in `gameOver`.
- Component sends a redundant `engine.send({ type: 'GAME_OVER' })` after
  `NEXT` already routes to `gameOver` via guard (which doesn't exist
  yet, per C1 тАФ so this is the only path that drives `gameOver` today).

Resolution direction: pick ONE path (guard-driven `NEXT тЖТ gameOver` OR
explicit `engine.send({ type: 'GAME_OVER' })`); add
`CELEBRATION_DONE` handler with `completeGame` action; decide whether
`gameOver` is `final` or regular (`final` blocks `RESUME_ROUND`
re-entry).

**C3 тАФ 22 stub assign actions block implementation**

- Bodies are `// Mirror reducer 'X' case` + `return context;` (no-op).
- Three concrete subtle bugs even if ported line-for-line: drag-state
  leak in `placeTile`, phantom `phase` field in `swapTiles`,
  ejectTile's early-return semantic lost.
- Plan Step 4 only checks state-name presence тАФ green tests + no-op
  assigns ship together.

Resolution direction: either inline full assign bodies (verbose but
explicit), OR add a Task 8.5 that ports
`answer-game-reducer.test.ts` cases relevant to NumberMatch into
`definition.test.ts` machine-level tests **before** Step 4 declares
pass.

**C4 тАФ `useGameRound` ├Ч XState seam asserted, not designed**

- Merged hook signature has no `dispatch` parameter and explicitly
  couples to a reducer (`UseGameRoundOptions<TRound>` and the spec's
  ┬зState ownership).
- Composition section claims hook callbacks "route to `engine.send`"
  for migrated games тАФ design contradiction.
- "Leave room (no exhaustive `as const` union locks)" instruction in
  PR 1a is unenforceable.

Resolution direction: write a "useGameRound ├Ч XState adapter" Spec
Delta on the merged spec (`docs/superpowers/specs/2026-05-03-use-game-round-design.md`)
prescribing: (a) a `dispatchTarget: 'reducer' | 'engine'` parameter
(or equivalent injection), (b) explicit kebabтЖТcamel phase translation,
(c) which side emits `round:resolved`, (d) what `currentRound` returns
when `engine.context.lastRoundOutput` is `{}`. The Spec Delta belongs
in the merged spec, not just in PR 1a's plan, because the merged spec
is the authoritative contract.
